Create `src/work1-chat-widget.test.ts` that tests through the real `<work1-chat-widget>` element. The widget uses ChatStore internally which uses ChatClient. Since component tests need to control the store state to verify DOM output, mock ChatClient (same pattern as chat-store.test.ts) so we can drive state via the store without needing a real WebSocket server.

**Test setup pattern:**
```typescript
import { describe, it, expect, beforeEach, afterEach, vi } from 'vitest';

// Mock ChatClient before importing widget
vi.mock('./chat-client.js', () => {
  class MockChatClient extends EventTarget {
    private _connected = false;
    get connected() { return this._connected; }
    _setConnected(v: boolean) { this._connected = v; }
    connect = vi.fn();
    disconnect = vi.fn();
    send = vi.fn();
  }
  return { ChatClient: MockChatClient };
});

// Dynamic import after mock
const { Work1ChatWidget } = await import('./work1-chat-widget.js');

function createWidget(attrs: Record<string, string> = {}): InstanceType<typeof Work1ChatWidget> {
  const el = document.createElement('work1-chat-widget');
  for (const [k, v] of Object.entries(attrs)) el.setAttribute(k, v);
  document.body.appendChild(el);
  return el as InstanceType<typeof Work1ChatWidget>;
}
```

**Access store for state manipulation:** The store is a private field. Access it via `(widget as any).store` for testing purposes. This is acceptable since we're testing the component from outside, driving internal state to verify DOM output.

**Helper to get mock client from store:**
```typescript
function getStoreClient(widget: any) {
  return widget.store?.client || (widget as any).store?.client;
}
function fireOnClient(widget: any, type: string, detail?: unknown) {
  const client = (widget as any).store.client;
  if (client) client.dispatchEvent(new CustomEvent(type, { detail }));
}
```

**Tests to implement (each in its own `it` block inside a `describe('UI states')`):**

1. **Initial/disconnected state** (no server-url, just mount):
   - Open the panel: call `(widget as any).store.toggleOpen()`; await updateComplete
   - Assert: `.input-textarea` exists and is disabled (`disabled === true`)
   - Assert: `.send-button` exists and is disabled

2. **Connected state**:
   - Set `server-url="ws://test"` attribute on widget
   - Simulate open: `(widget as any).handleOpen()`; await updateComplete
   - This triggers connect. Fire connected event on the mock client: `fireOnClient(widget, 'connected', { session_id: 'test-123' })`
   - await updateComplete
   - Assert: `.input-textarea` exists and is NOT disabled
   - Assert: `.send-button` exists and is NOT disabled
   - Assert: no `.message--error` or `.message--system` visible (no error banners)

3. **Streaming state**:
   - Start from connected state (as above)
   - Fire token event: `fireOnClient(widget, 'token', { content: 'Hello' })`
   - await updateComplete
   - Assert: a message element with streaming content exists. Look for `.message--streaming` or a message with `.message--agent` that contains 'Hello'

4. **Error state**:
   - Start from connected state
   - Fire error event: `fireOnClient(widget, 'error', { content: 'Something went wrong' })`
   - await updateComplete
   - Assert: a `.message--system` or `.message--error` element exists containing 'Something went wrong'

5. **Session ended state**:
   - Start from connected state
   - Fire session_end event: `fireOnClient(widget, 'session_end', { reason: 'timeout', content: 'Session timed out' })`
   - await updateComplete
   - Assert: `.input-textarea` is disabled
   - Assert: a system message containing 'timeout' exists
   - Assert: `.new-conversation-button` is visible (since eventsWired=true, connectionState=disconnected, messages.length>0)

**Important notes:**
- Skip CSS theming tests per user decision (happy-dom has no real CSS engine)
- Focus on DOM structure and attribute presence only
- Use `widget.shadowRoot!.querySelector()` and `widget.shadowRoot!.querySelectorAll()` for all queries
- Always `await (widget as any).updateComplete` after state changes before asserting
- In `afterEach`, call `widget.remove()` to clean up
- If happy-dom doesn't support certain shadow DOM features, document the limitation and mark the specific assertion with a comment, but still include the test

**Happy-DOM timing note:** After `fireOnClient()`, the store calls `host.requestUpdate()` which schedules a Lit render. Always await `updateComplete` before querying DOM. If assertions fail due to timing, try `await new Promise(r => setTimeout(r, 0))` then `await widget.updateComplete` again.
