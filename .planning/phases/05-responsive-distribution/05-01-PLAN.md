**panel.styles.ts** -- Add at the end of the css template literal, AFTER the existing rules:

```css
@media (max-width: 480px) {
  .chat-panel {
    position: fixed;
    inset: 0;
    width: 100%;
    height: 100vh;
    height: 100dvh;
    border-radius: 0;
    box-shadow: none;
    bottom: 0;

    left: 0;
  }

```

Order matters: `100dvh` AFTER `100vh` so browsers supporting dvh override the fallback. The panel overrides width/height CSS custom properties by using `100%` and `100dvh` directly (fixed values beat var() on mobile). safe-area-inset-* padding protects against notch and home indicator.

**widget.styles.ts** -- Add at the end of the css template literal:
```css
@media (max-width: 480px) {
  .bubble--hidden-mobile {
    transform: scale(0);

    pointer-events: none;
  }
}
```

This class will be applied to the bubble button when the panel is open AND viewport is mobile. The existing `.bubble--hidden` class is toggled in JS; this new class is purely CSS-driven via the media query, but still needs to be applied conditionally (when panel is open). Since we cannot conditionally add the class only on mobile from JS without a resize listener, use a simpler approach: always add a `bubble--hidden-mobile` class when panel is open (same condition as `bubble--hidden`), and the @media query controls whether the rule actually hides it. Actually, the bubble is ALREADY hidden via `bubble--hidden` when panel is open (from Phase 2). On mobile, the bubble should remain hidden. This already works.   <name>Task 2: Add visualViewport keyboard handler to widget</name>
  <files>src/work1-chat-widget.ts</files>
  <action>
Add a visualViewport resize handler to Work1ChatWidget per user decision to detect virtual keyboard and resize the panel.

Add a private field to track the handler for cleanup:
```typescript
private viewportHandler: (() => void) | null = null;
```

Add a method `setupKeyboardHandler()` called from `handleOpen()` (after panel opens):
```typescript
private setupKeyboardHandler(): void {
  if (!window.visualViewport || this.viewportHandler) return;

  this.viewportHandler = () => {
    const vv = window.visualViewport!;
    const keyboardHeight = window.innerHeight - vv.height;
    const panel = this.renderRoot.querySelector('.chat-panel') as HTMLElement | null;
    if (!panel) return;

    if (keyboardHeight > 100) {
      // Virtual keyboard visible -- shrink panel to visible area
      panel.style.height = `${vv.height}px`;
    } else {
      // Keyboard dismissed -- clear inline override, CSS takes over
      panel.style.height = '';
    }
  };

  window.visualViewport.addEventListener('resize', this.viewportHandler);
}
```

Add cleanup in a `teardownKeyboardHandler()` method called from `disconnectedCallback()`:
```typescript
private teardownKeyboardHandler(): void {
  if (window.visualViewport && this.viewportHandler) {
    window.visualViewport.removeEventListener('resize', this.viewportHandler);
    this.viewportHandler = null;
  }
}
```

Update `disconnectedCallback()` to call `this.teardownKeyboardHandler()` before `super.disconnectedCallback()`.

Call `this.setupKeyboardHandler()` at the end of `handleOpen()`, right after the existing connect/wire logic. The handler is safe to call multiple times (guards with `if (this.viewportHandler) return`).

No need for a mobile-only guard on this handler -- on desktop, `window.innerHeight - vv.height` will be ~0, so the keyboard branch never triggers. The handler is harmless on desktop.
