<task type="auto">
  <name>Task 2: Create publish workflow for npm releases with provenance</name>
  <files>.github/workflows/publish.yml</files>
  <action>
Create the npm publish workflow file.

publish.yml specification:
- name: "Publish"
- Trigger: `release` event, `types: [published]`
- Single job: `publish` on `ubuntu-latest`
- Environment: `npm` (for protection rules)
- Permissions: `contents: read`, `id-token: write` (REQUIRED for provenance)
- Steps:
  1. `actions/checkout@v4`
  2. `actions/setup-node@v4` with `node-version: 20`, `cache: npm`, and `registry-url: https://registry.npmjs.org/` (CRITICAL: provenance breaks without explicit registry-url)
  3. `npm ci`
  4. Version verification step named "Verify version matches tag":
     ```
     PACKAGE_VERSION=$(node -p "require('./package.json').version")
     TAG_VERSION="${GITHUB_REF_NAME#v}"
     if [ "$PACKAGE_VERSION" != "$TAG_VERSION" ]; then
       echo "::error::package.json version ($PACKAGE_VERSION) does not match tag ($TAG_VERSION)"
       exit 1
     fi
     ```
  5. `npm run build`
  6. `npm test`
  7. `npm publish --provenance --access public` with env `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}`

Self-contained per user decision: runs its own build + test, does NOT rely on CI workflow.
Uses manual version management: developer bumps package.json, creates release with matching vX.Y.Z tag, workflow verifies match.
  </action>
  <verify>
    <automated>node -e "const yaml = require('yaml'); const fs = require('fs'); const doc = yaml.parse(fs.readFileSync('.github/workflows/publish.yml','utf8')); const perms = doc.jobs.publish.permissions; if (!perms['id-token'] || perms['id-token'] !== 'write') throw new Error('Missing id-token: write permission'); console.log('Valid: has id-token write permission')"</automated>
  </verify>
  <done>publish.yml triggers on GitHub Release, verifies version match, runs build+test, publishes with --provenance --access public, uses npm environment and NPM_TOKEN secret</done>
</task>

  <files>.github/workflows/docs.yml</files>
  <action>
Create the VitePress docs deployment workflow file.

docs.yml specification:
- name: "Deploy Docs"
- Triggers: push to `master` branch, `workflow_dispatch` (manual trigger)
- Top-level permissions: `contents: read`, `pages: write`, `id-token: write`
- Concurrency group: `group: pages`, `cancel-in-progress: false` (prevents overlapping deployments, recommended by VitePress official docs)
- Two jobs:

Job 1 - `build`:
  - runs-on: `ubuntu-latest`
  - Steps:
    1. `actions/checkout@v4`
    2. `actions/setup-node@v4` with `node-version: 20` and `cache: npm`
    3. `actions/configure-pages@v4`
    4. `npm ci`
    5. `npm run docs:build`
    6. `actions/upload-pages-artifact@v3` with `path: docs/.vitepress/dist`

Job 2 - `deploy`:
  - needs: `build`
  - runs-on: `ubuntu-latest`
  - environment: `name: github-pages`, `url: ${{ steps.deployment.outputs.page_url }}`
  - Single step: `actions/deploy-pages@v4` with `id: deployment`, named "Deploy to GitHub Pages"

No path filtering per user decision: deploys on every push to master.
Uses modern actions/deploy-pages approach per user decision: no gh-pages branch.
  </action>
  <verify>
    <automated>node -e "const yaml = require('yaml'); const fs = require('fs'); const doc = yaml.parse(fs.readFileSync('.github/workflows/docs.yml','utf8')); if (!doc.jobs.build || !doc.jobs.deploy) throw new Error('Missing build or deploy job'); if (!doc.jobs.deploy.needs) throw new Error('Deploy must depend on build'); console.log('Valid: has build and deploy jobs with dependency')"</automated>
  </verify>
  <done>docs.yml triggers on push to master, builds VitePress docs, deploys to GitHub Pages using actions/deploy-pages, includes concurrency group</done>
