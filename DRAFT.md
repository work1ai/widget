# Work1 Front End Chat Widget

## Description
This widget is the gateway to stream chats (websockets) with a Agent Chat backend. It must expose a Widget class in React, Vue, Angular, or Vanilla JS, etc, front-end applications that instantiate a chat window that is configurable and customizable by the customer (the owner of the front-end application) and connect with the Agent Chat backend.
- The connection is established and persisted while the chat window is active.
- The chat widget allows for streaming messages.
- The user can configure the chat window width, colors, initial message, position on screen, icon, etc.
- This widget can be used nativelly in React, Vue, Angular, 

---

# Widget Interface Contract

**Version:** 1.0 (matches chat-server v0.1.0)
**Protocol:** WebSocket over TLS (`wss://`)
**Endpoint:** `/ws`

---

## 1. Connection

### Endpoint

```
wss://<chat-server-host>/ws
```

### Required Headers

| Header | Required | Notes |
|--------|----------|-------|
| `Origin` | **Yes** | Must exactly match one of the server's `ALLOWED_ORIGINS`. Connection is rejected without it. |

### Connection Limits

| Constraint | Default | Behavior on Violation |
|------------|---------|----------------------|
| Max concurrent connections per IP | 5 | WebSocket closed with code `1008` (Policy Violation) **before accept** — no error event sent |
| Origin not in allowlist | — | WebSocket closed with code `1008` **before accept** — no error event sent |

### Successful Connection

On accept, the server immediately sends:

```json
{"type": "connected", "session_id": "<uuid-v4>"}
```

The `session_id` is the canonical identifier for this conversation session.

---

## 2. Messages — Client to Server

The widget sends exactly **one message type**:

```json
{"type": "message", "content": "<string>"}
```

| Field | Type | Constraints |
|-------|------|-------------|
| `type` | `"message"` (literal) | Must be exactly this string |
| `content` | `string` | Non-empty text |

### Size Limit

- **Default max:** 4096 bytes (total JSON payload)
- If exceeded, the server responds with an error event **but keeps the connection open**:
  ```json
  {"type": "error", "content": "Message too large. Maximum size is 4096 characters."}
  ```
- The message is **not** forwarded to the agent.

---

## 3. Messages — Server to Client

### 3.1 Gateway Events (generated by chat-server)

| Event | Schema | When |
|-------|--------|------|
| `connected` | `{"type":"connected","session_id":"<uuid>"}` | Immediately after WebSocket accept |
| `status` | `{"type":"status","content":"<string>"}` | Agent is using a tool, or connection restored after reconnect |
| `reconnecting` | `{"type":"reconnecting"}` | Agent connection dropped, server is retrying |
| `session_end` | `{"type":"session_end","reason":"<reason>","content":"<message>"}` | Session terminated |
| `error` | `{"type":"error","content":"<string>"}` | Recoverable error (connection stays open) or fatal error (connection closes after) |

**`status` content values from tool use:**

| Tool | Status Message |
|------|---------------|
| `pull_service_facts` | `"Looking up service details..."` |
| `dispatch_message` | `"Sending your request..."` |
| Unknown tool | `"Processing..."` |

**`session_end` reasons:**

| Reason | Content | Trigger |
|--------|---------|---------|
| `"idle_timeout"` | `"Session closed due to inactivity."` | No messages for 300 seconds (default) |

### 3.2 Agent Events (pass-through from agent server)

| Event | Schema | Purpose |
|-------|--------|---------|
| `token` | `{"type":"token","content":"<string>"}` | Streamed text chunk (character-by-character) |
| `typing` | `{"type":"typing","active":true\|false}` | Agent typing indicator |
| `message_end` | `{"type":"message_end"}` | End of a complete agent response |
| `session_start` | `{"type":"session_start","session_id":"<string>"}` | Agent session initialized |
| `error` | `{"type":"error","content":"<string>"}` | Agent-side error |

`tool_use` events are **never** forwarded to the client. They are transformed into `status` events (see 3.1).

---

## 4. Connection Lifecycle

```
1. Client opens WebSocket to wss://<host>/ws with Origin header
2. Server validates origin + rate limit
3. Server accepts → sends {"type":"connected","session_id":"..."}
4. Server connects internally to agent server
5. Bidirectional proxy active:
   - Client sends: {"type":"message","content":"..."}
   - Server sends: token, typing, message_end, status, error events
6. Session ends via:
   a. Client closes WebSocket (graceful)
   b. Idle timeout → server sends session_end then closes
   c. Agent unreachable → server sends error then closes
```

### Reconnection Behavior (server-side, agent connection)

When the agent connection drops mid-session:

1. Server sends `{"type":"reconnecting"}` to client
2. Server retries up to **3 times** with exponential backoff (1s, 2s, 4s)
3. During retries, client messages are **buffered** (up to 50 messages)
4. If reconnect succeeds: `{"type":"status","content":"Connection restored."}`
5. If reconnect fails: `{"type":"error","content":"Connection to our service was lost and could not be restored. Please start a new conversation."}` then connection closed

### Keep-Alive

- The server does **not** send application-level pings to clients
- Standard WebSocket protocol pings are handled automatically
- The widget should handle its own keep-alive if needed (browser WebSocket APIs do this automatically)
- **Idle timeout** (300s default) is based on application messages, not pings

---

## 5. Error Scenarios

| Scenario | What Client Receives | Connection |
|----------|---------------------|------------|
| Invalid origin | Nothing (close code `1008`) | Rejected before accept |
| Rate limit exceeded | Nothing (close code `1008`) | Rejected before accept |
| Message too large | `{"type":"error","content":"Message too large..."}` | Stays open |
| Agent unreachable on connect | `{"type":"error","content":"Our service is temporarily unavailable..."}` | Closed after |
| Agent drops mid-session | `{"type":"reconnecting"}` then success or error | Depends on reconnect |
| Idle timeout | `{"type":"session_end","reason":"idle_timeout",...}` | Closed after |

### WebSocket Close Codes

| Code | Meaning | When |
|------|---------|------|
| `1000` | Normal closure | Graceful disconnect |
| `1008` | Policy violation | Origin rejected or rate limited |
| `1011` | Server error | Unexpected exception |

## 6. Widget Implementation Requirements

Based on the contract above, the widget must:

### Connection

- [ ] Connect to `wss://<host>/ws` — the `Origin` header is set automatically by the browser
- [ ] Handle connection rejection (WebSocket `onerror`/`onclose` with code `1008`) — show "unable to connect" UI
- [ ] Store the `session_id` from the `connected` event for display/debugging

### Sending Messages

- [ ] Send JSON: `{"type":"message","content":"<text>"}`
- [ ] Enforce client-side max length of 4096 bytes before sending (defensive)
- [ ] Disable send button while no active WebSocket connection

### Receiving Messages — Rendering

- [ ] **`token`** — Append content to current message bubble (streaming effect)
- [ ] **`typing` (active=true)** — Show typing indicator
- [ ] **`typing` (active=false)** — Hide typing indicator
- [ ] **`message_end`** — Finalize current message bubble
- [ ] **`status`** — Show as a transient system/status indicator (e.g., "Looking up service details...")
- [ ] **`connected`** — Transition UI from "connecting" to "ready" state
- [ ] **`reconnecting`** — Show "reconnecting..." banner/indicator
- [ ] **`session_end`** — Show reason message, disable input, offer "Start new conversation" action
- [ ] **`error`** — Display error message; if connection closes after, transition to disconnected state

### Resilience

- [ ] Handle `reconnecting` — show non-blocking indicator, keep message history visible
- [ ] Handle unexpected WebSocket close — offer reconnection UI
- [ ] Buffer user input during `reconnecting` state (server buffers up to 50 messages, but the connection to the client stays open)
- [ ] Handle malformed/unknown event types gracefully (ignore or log)

### Privacy

- [ ] The conversation is with an AI system — this should be disclosed in the widget UI
- [ ] Conversation data is ephemeral (not persisted after session ends)
- [ ] Pre-book packet data (name, phone, email, address) is retained by the service team

## 7. Health Check (Optional)

```
GET https://<chat-server-host>/health
```

Response:

```json
{
  "status": "ok | degraded",
  "agent": "ok | unreachable",
  "active_sessions": 12,
  "uptime_seconds": 3600
}
```

The widget can optionally poll this before showing the chat button to avoid offering chat when the service is degraded.
